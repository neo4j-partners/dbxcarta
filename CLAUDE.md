# DBxCarta — CLAUDE.md

## Critical design rules

The Spark ingest pipeline that reads Unity Catalog and builds the Neo4j semantic layer now lives in the external [neocarta](https://github.com/neo4j-field/neocarta) connector, not here. dbxcarta is the operator tooling that stages and runs that connector wheel, plus the local evaluation client (`dbxcarta-client`, plain local Python) and the `materialize` job. Do not reintroduce a Spark ingest pipeline in this repo; pipeline logic changes go to neocarta.

`docs/reference/best-practices.md` is the authoritative list of design rules with cited sources. It documents both the neocarta ingest pipeline's rules (so you understand the pipeline dbxcarta runs) and the rules dbxcarta still owns (operator submit, the config boundary, the client's node-identity handling, the materialize run-summary, the submit bootstrap smoke check). Read it before changing any Spark or Spark-adjacent code (the `materialize` job, the submit tooling) or the client; add to it (with a cited source) when a rule is established, and do not let these rules regress. In particular:

- **Never use a Python UDF or `pandas_udf` for rule logic in a Spark pipeline.** Catalyst cannot optimize a UDF: it blocks predicate pushdown and whole-stage codegen and pays Arrow serialization per batch, which dominates large or n²-shaped joins. Logic driven by small static tables (suffix lists, type-class maps, score tables, stopwords) must be expanded into native `Column` expressions and broadcast-join lookups at plan-construction time. Python builds the plan; Spark evaluates every row. This governs the neocarta ingest pipeline (`docs/reference/best-practices.md` §7, Spark / Databricks); honor it in any Spark code here too.
- **Never collect catalog-scale data to the driver.** Keep columns, constraints, values, and FK candidates in DataFrames. See `docs/reference/best-practices.md` §5.

## Environment configuration (layered)

dbxcarta loads env config in two layers. See `docs/proposals/env-layering.md`.

- The repo-root `.env` is the shared **base**: Databricks infra (profile, compute, workspace, warehouse) and the shared chat/embedding endpoints. It is gitignored and **never edited per integration**. It does **not** set `DATABRICKS_SECRET_SCOPE` (that is per-integration, see below) and it holds **no `NEO4J_*` secrets** — those are per-integration and live in each example's standalone `.env` so the two cannot drift.
- Each integration ships a committed, secret-free `examples/<name>/dbxcarta-overlay.env`. This is the **single source of truth** for that integration's per-example dbxcarta config: catalog/catalogs/layer-map/schemas, volume, summary, sample/embedding flags, client arms, **and the per-integration `DATABRICKS_SECRET_SCOPE`** (a scope name, not a secret). Readiness is driven by this overlay through `dbxcarta ready` (backed by `dbxcarta.core.readiness`), and the local `dbxcarta-client` reads the example's bundled `questions.json` straight off local disk; there is no question-upload step and no per-example preset object.
- Select an integration with one flag: `uv run dbxcarta <cmd> --env-file examples/<name>/dbxcarta-overlay.env`, or export `DBXCARTA_ENV_FILE` to that path. `--env-file` wins over `DBXCARTA_ENV_FILE`. The submit path forwards the overlay's `KEY=VALUE` pairs to the cluster as job parameters, so the overlay must stay secret-free.
- Precedence, lowest to highest: base `.env` → overlay → real exported process env. A selected overlay that does not resolve is a hard error, never a silent base-only fallback. `databricks_secret_scope` has no code default, so a run with no overlay fails loudly at config load.
- Do not put `NEO4J_*` or any secret in an overlay (overlays are committed and shipped as job params). Do not write integration-scoped values back into the root `.env`.
- The `examples/<name>/.env` and `.env.sample` files are the **separate** self-contained config for that integration's standalone tooling (local demo, slice/materialize) and hold the per-integration `NEO4J_*` secrets. `setup_secrets.sh` reads the scope name from the overlay and the `NEO4J_*` values from this `.env` to provision the Databricks secret scope. This file as a whole never layers and is not the dbxcarta CLI overlay; the **one** exception is that off-cluster the `dbxcarta-client` reads **only** the `NEO4J_*` keys from the selected overlay's sibling `.env` (via `load_overlay_secrets`), since the committed overlay cannot carry secrets and on-cluster jobs get them from the secret scope. None of the file's other keys (its own catalog, volume, summary table) ever reach the dbxcarta environment.
